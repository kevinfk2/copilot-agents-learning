# 并行执行模式（Fleet Mode）

## 概述

Fleet Mode 让你可以**同时启动多个子代理**来并行处理任务，显著提高复杂工作流的效率。

## 启用 Fleet Mode

```bash
/fleet              # 启用并行子代理模式
```

## 核心概念

### 单 Agent 模式（默认）
```
主 Agent 
  └─ 子任务 1
     └─ 子任务 2
        └─ 子任务 3
```
总耗时 = T1 + T2 + T3

### Fleet Mode（并行模式）
```
主 Agent
  ├─ 子任务 1 ┐
  ├─ 子任务 2 ├─ 并行执行
  └─ 子任务 3 ┘
```
总耗时 ≈ max(T1, T2, T3)

## 使用场景

### ✅ 适合 Fleet Mode 的场景

1. **多文件独立分析**
   - 分析 5 个不同的模块
   - 检查 10 个不同的配置文件
   - 审查 3 个不同 PR 的部分

2. **并行数据处理**
   - 获取多个数据源的信息
   - 运行多个独立测试套件
   - 执行多个无依赖的数据转换

3. **并行代码生成**
   - 为 3 个不同的 API 端点生成代码
   - 创建多个相关的新文件
   - 实现多个独立的功能模块

4. **研究和探索**
   - 在 5 个不同的代码库中搜索相同的模式
   - 调查 3 个不同的设计方案
   - 收集多个独立的学习资源

### ❌ 不适合 Fleet Mode 的场景

- 任务有顺序依赖（A 必须完成才能做 B）
- 任务之间需要频繁通信或共享状态
- 单个任务时间很短（<5秒）
- 任务总数 > 10 个（会过载）

## 实践例子

### 例子 1：并行分析多个模块

```bash
# 需求：分析前端、后端、数据库三个模块的架构

# Fleet Mode 实现
启用 /fleet，然后要求：
"请同时分析以下三个模块的架构：
1. frontend/ - React 组件结构
2. backend/ - API 设计
3. database/ - 数据模型"

# 结果：三个分析同时进行，总时间减少 60%
```

### 例子 2：并行代码审查

```bash
# 需求：审查 PR 中的 3 个独立功能变更

# Fleet Mode 实现
启用 /fleet，分别委派给 Code Review Agent：
- 审查 auth 相关变更
- 审查 API 端点变更
- 审查数据库迁移脚本

# 结果：3 个审查同时进行
```

### 例子 3：并行测试运行

```bash
# 需求：运行 unit、integration、e2e 三种测试

# Fleet Mode 实现
启用 /fleet，然后要求 Task Agent：
- 运行 npm run test:unit
- 运行 npm run test:integration
- 运行 npm run test:e2e

# 结果：三个测试套件同时运行
```

## 最佳实践

### 1. 明确任务边界
```bash
✅ 好：
"Agent 1: 分析 users 模块
 Agent 2: 分析 products 模块
 Agent 3: 分析 orders 模块"

❌ 不好：
"分析所有模块的关系"（模糊的并行化）
```

### 2. 提供完整上下文
```bash
# 给每个子代理足够的信息
"Task: 审查 file1.ts 的变更
Context: 这是用户认证模块
Look for: SQL 注入漏洞、错误处理"
```

### 3. 监控任务进度
```bash
/tasks              # 查看所有运行中的任务
```

### 4. 设置合理数量
```
最优范围：3-8 个并行任务
过少（1-2）：浪费 Fleet Mode 的优势
过多（>10）：系统过载，反而拖慢速度
```

### 5. 任务后处理
```bash
# Fleet Mode 完成后，你可能需要：
1. 汇总结果
2. 解决冲突
3. 生成最终报告
```

## 性能优化技巧

### 1. 任务粒度平衡
```bash
❌ 不好 - 粒度太细：
Agent 1: 检查变量 x
Agent 2: 检查变量 y
Agent 3: 检查变量 z
（太多任务，管理开销大）

✅ 好 - 平衡的粒度：
Agent 1: 审查数据层逻辑
Agent 2: 审查业务逻辑
Agent 3: 审查 API 层逻辑
```

### 2. 预测单任务时间
```bash
如果每个任务需要 30-90 秒，Fleet Mode 最有效
- 太快的任务：创建和同步的开销可能超过收益
- 太慢的任务：可能一个就用完超时时间
```

### 3. 批量任务管理
```bash
# 如果有 15 个任务需要并行处理
第一批：/fleet 运行 6-8 个
等待完成 → 第二批：运行剩余的
```

## 故障排除

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 某个子代理失败 | 任务定义不清 | 提供更详细的上下文 |
| 所有代理超时 | 任务太复杂 | 分解为更小的子任务 |
| 结果不一致 | 代理之间有依赖 | 改为顺序执行 |
| 结果整合困难 | 输出格式不统一 | 要求所有代理统一输出格式 |
